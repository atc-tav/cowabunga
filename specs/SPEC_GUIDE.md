# What a good spec looks like (and how to lint one)

> A spec is the **raw material** the factory consumes. It does not, by itself,
> make a faithful game — all four of our audited games had rich specs and three
> still came out fake (see
> [`docs/spec-driven-development.md`](../docs/spec-driven-development.md)). But a
> *bad* spec makes a faithful game **impossible**: you cannot build an Oracle
> Ledger out of vagueness, and you will faithfully implement a contradiction.
>
> This doc defines the quality bar a spec must clear **before** anyone builds an
> Oracle Ledger from it, and the lint checklist to confirm it.

## The test of a good spec

> **Every claim in the spec can be turned into an automatable assertion, or is
> explicitly flagged as human-only.**

If a sentence can't become a ledger row (a unit check, a scenario, an
invariant) or an honest "human judges this," it is decoration — tighten it or
cut it. The spec exists to be *executed*, not admired.

## What every spec must contain

A spec good enough to build from has all of these. Our existing specs
(`arkanoid.md`, `galaga-nes.md`, `donkey-kong-nes.md`, `mario-bros-arcade.md`)
are the reference shape — copy their structure.

1. **Source & platform header** — which exact original (arcade vs. NES, year,
   maker). Faithfulness is *to a specific machine*; "Galaga" arcade ≠ NES.
2. **Screen / resolution** — native pixel dimensions and orientation. (Galaga
   shipped at the wrong resolution because this got silently changed — a row
   that should have caught it.)
3. **A constants table** — every speed, timing, dimension, count, threshold,
   with a name and a value. No magic numbers left in prose. These become
   `constants.ts` *and* unit-test oracles verbatim.
4. **A scoring table** — every point value, every multiplier, every bonus, every
   extra-life threshold. (Mario Bros. scored every value wrong; a scoring table
   with assertions makes that impossible.)
5. **Every mechanic, enumerated** — movement, collision, enemy AI/behavior,
   power-ups, hazards, win/lose conditions. One mechanic per heading.
6. **Every stage / level / phase** — explicitly listed. (Donkey Kong shipped 2
   of 3 stages; an enumerated stage list with a row per stage makes a missing
   stage a glaring red row, not a silent omission.)
7. **Sprites** — described well enough to draw programmatically (ASCII grids,
   palettes, per-state variants). Note state-dependent recolors (enraged, last
   enemy, captured, damaged).
8. **`✅ CHECK:` callouts** — embedded self-verification notes. **Each one must
   be phrased so it maps to a single automatable assertion** (or is marked
   human-only). These seed the P0/P1 ledger rows directly.

## The spec-lint checklist (Step 0 of the pipeline)

Run this against `specs/<game>.md` **before** building the Oracle Ledger. Any
unchecked box is a question for the human, not a guess.

- [ ] **No internal contradictions.** The prose, the tables, and the CHECK
      callouts agree with each other. *(Known offenders: the DK spec disagrees
      with itself about which stages exist; the Arkanoid spec's silver-hit prose
      contradicts its own hit table. Both must be ruled on by a human.)*
- [ ] **Every constant has a value** — no "fast", "a few", "short delay" without
      a number.
- [ ] **Every scoring event has a point value**, including multipliers and
      extra-life thresholds.
- [ ] **Every stage/level/phase is enumerated and counted** — and the count is
      stated once, unambiguously.
- [ ] **Every `✅ CHECK` is automatable** — or explicitly tagged human-only.
- [ ] **No mechanic is described only by analogy** ("like the arcade version")
      without the actual rule.
- [ ] **Sprite states are complete** — every visual variant a state can take is
      listed.
- [ ] **No mechanic requires a shared primitive we don't have** without that
      being called out as a build dependency.
- [ ] **Delta specs name their base.** (The NES Dig Dug spec is a delta on the
      arcade spec — read the base first; every `⚠️ NES DIFF:` is a ledger row.)

**If the spec fails lint:** do not start building. Surface the specific failures
to the human via `AskUserQuestion`, propose a resolution, and amend the spec.
Fixing the spec is cheaper than faithfully implementing its bugs.

## Authoring or adding a spec

Drop the markdown here with a lowercase-kebab filename (e.g.
`pac-man-arcade.md`) and add a row to the index in
[`README.md`](./README.md). New specs should clear this guide's bar on day one.

> **Missing today:** Pac-Man has no spec (it predates this folder). A clone
> without a spec cannot be scored for faithfulness or regenerated by the
> factory — it's flying blind. Back-filling a `pac-man-arcade.md` that clears
> this bar is the prerequisite to ever holding Pac-Man to the same standard as
> Arkanoid.
